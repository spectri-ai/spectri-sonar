# Using Spectri

This project uses the Spectri artifact system. You MUST follow these rules.

## The Rule

<EXTREMELY_IMPORTANT>
Before performing ANY task, check if a Spectri skill or script exists for it.

If there is even a 1% chance a skill might apply, you MUST invoke it BEFORE doing anything else. If the skill turns out to be wrong for the situation, you don't need to use it — but you must check first.

This is not negotiable. This is not optional. You cannot rationalise your way out of this.
</EXTREMELY_IMPORTANT>

## How to Check

1. Check the skill and command catalogs below
2. If a skill matches, invoke it via the Skill tool
3. If a `/spec.*` command matches, run it
4. If a script exists for the mechanical part, run it
5. Only proceed manually if NO skill, NO command, and NO script exists — and state why

## Work Cycle

Every piece of work follows a workflow skill. Before writing any code or creating any artifact, identify which workflow applies:

- **Code Change** — ad-hoc changes with no issue or task
- **Task Implementation** — implementing from tasks.md
- **Issue Resolution** — fixing a tracked issue

If none applies, you are doing something wrong. Stop and ask.

## Commit Bundle

Every commit is a complete bundle. Before committing, verify ALL of these are staged:

- The code or artifact change itself
- Spec update (if the change affects behaviour described by a spec)
- Resolved issue with `## Resolution` filled in (if resolving an issue)
- Implementation summary (run `create-implementation-summary.sh`)

Committing code alone — without spec update, tests, and implementation summary — is the most common agent failure pattern.

## Iron Laws

- **Never fabricate timestamps.** Always call `get-timestamp.sh`.
- **Never edit meta.json manually.** Always use `update-spec-meta.sh`.
- **Never create artifacts manually** when a creation script or command exists.
- **Never skip the work cycle.** Every code change goes through a workflow skill.

## Three Types of Automation

| Type | What it does | Your obligation |
|------|-------------|-----------------|
| **Skills** | Orchestrate work — guide you through a workflow | Load the SKILL.md and follow it |
| **Scripts** | Execute mechanically — create files, update metadata, validate | Run the script. Do not replicate what it does manually. |
| **Hooks** | Guard and constrain — fire automatically on tool events | You don't call these. They call you. If a hook blocks you, comply — do not work around it. |

## Red Flags — Rationalisations That Mean You Are About to Skip a Skill

| You're thinking... | The truth |
|--------------------|-----------|
| "This is just a simple question" | Questions are tasks. Check for skills. |
| "I know how to do this" | Knowing how is not the same as following the workflow. Check. |
| "I need more context first" | Skill check comes BEFORE gathering context. Skills tell you HOW to gather context. |
| "Let me explore the codebase first" | Skills tell you HOW to explore. Check first. |
| "It's just a quick fix" | Quick fixes are the most common source of skipped workflows. Check. |
| "I'll use the skill after I understand the problem" | No. The skill IS how you understand the problem. |
| "The skill is too heavyweight for this" | You haven't loaded it yet. You don't know that. Load it and find out. |

## Command Catalog

Commands are explicit user-invoked workflows via `/spec.*`. Check these alongside skills.

### Core
- `/spec.specify` — Create or update a feature specification
- `/spec.plan` — Generate implementation plan from spec
- `/spec.tasks` — Generate tasks from plan
- `/spec.implement` — Execute tasks from tasks.md
- `/spec.checklist` — Generate custom checklist for a feature
- `/spec.retro` — Document an already-implemented feature

### Modification
- `/spec.revise-spec` — Refine user stories, FRs, acceptance criteria
- `/spec.revise-plan` — Refine implementation approach
- `/spec.extend-tasks` — Add tasks to align with plan changes
- `/spec.clarify` — Ask targeted questions, encode answers into spec

### Quality
- `/spec.validate` — Check format, sections, consistency
- `/spec.cross-check-alignment` — Cross-artifact consistency analysis
- `/spec.health` — Detect structural drift in the project
- `/spec.issue` — Create issue from rapid capture
- `/spec.taskstoissues` — Convert tasks to GitHub issues

### Trail
- `/spec.summary` — Create implementation summary
- `/spec.rfc` — Create or update RFC
- `/spec.adr` — Create ADR
- `/spec.constitution` — Create or update constitution
- `/spec.update-meta` — Update spec meta.json fields
